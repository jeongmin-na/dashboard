## 🚀 시작하기

### 시스템 요구사항
- 웹 브라우저 (Chrome, Firefox, Safari, Edge)
- 인터넷 연결
- Python 3.7 이상 (프록시 서버용)

### 설치 및 실행

1. **프록시 서버 시작**
   ```bash
   python proxy_server.py
   ```
   - 서버가 `localhost:8001`에서 실행됩니다
   - 콘솔에 접속 URL이 표시됩니다

2. **대시보드 접속**
   - 브라우저에서 `http://localhost:8001/dash.html` 접속
   - 자동으로 Overview 섹션이 표시됩니다

## 📈 Overview 섹션

### 통계 카드
- **전체 멤버**: 팀 전체 멤버 수
- **활성 멤버**: 선택된 기간 내 활동한 멤버 수
- **비활성 멤버**: 선택된 기간 내 활동하지 않은 멤버 수

### 사용량 개요 카드
- **Total Users**: 선택된 기간 내 실제 활동한 사용자 수
- **Lines of Agent Edits**: 선택된 기간 내 에이전트 편집 라인 수
- **Tabs Accepted**: 선택된 기간 내 수락된 탭 수
- **Chats**: 선택된 기간 내 채팅 요청 수

> **중요**: 사용량 개요 카드는 Overview 섹션의 기간 필터(1d, 7d, 30d, 사용자 정의)를 정확히 반영합니다.

### 기간 필터
- **1d**: 최근 1일
- **7d**: 최근 7일
- **30d**: 최근 30일
- **사용자 정의**: 특정 시작일과 종료일 선택

> **새로운 기능**: 기간 필터 왼쪽에 선택된 날짜 범위가 표시됩니다.
> - 예: 1d 선택 시 "2025-08-02 ~ 2025-08-03" (오늘 기준 1일 전)
> - 예: 7d 선택 시 "2025-07-27 ~ 2025-08-03" (오늘 기준 7일 전)
> - 사용자 정의 날짜 선택 시에도 선택한 범위가 정확히 표시됩니다.
> - **한국 시간 기준**: 모든 날짜는 한국 시간대(UTC+9)를 기준으로 계산됩니다.
> - **즉시 반응**: 버튼 클릭과 동시에 날짜 범위가 즉시 업데이트됩니다.

### 활동 차트
- **Y축**: 활동 수량
- **X축**: 날짜 (선택된 기간 필터 기준)
- **동적 업데이트**: 기간 필터 선택 시 차트 X축도 함께 변경
- **상태 유지**: 다른 메뉴로 이동 후 Overview로 돌아와도 선택된 필터 상태 유지
- **데이터 패턴**: 
  - 높은 피크: 대규모 프로젝트나 집중 작업 기간
  - 낮은 구간: 휴가나 주말 기간
  - 점진적 증가: 지속적인 프로젝트 진행

### 활동 리스트
- **사용자 이메일**: 클릭하여 상세 정보 확인
- **마지막 활동**: 최근 활동 시간
- **Premium Requests 수**: Fast Requests 사용 횟수
- **지출**: 달러 단위로 표시
- **한도 초과**: 설정된 한도 초과 여부

### 페이지네이션 기능
- 한 페이지당 10개 항목 표시
- 페이지 번호 클릭으로 이동
- 이전/다음 페이지 버튼

## 🔧 문제 해결

### 일반적인 문제

**Q: 대시보드에 접속할 수 없습니다**
A: 프록시 서버가 실행 중인지 확인하세요. `python proxy_server.py` 명령어로 서버를 시작하세요.

**Q: 데이터가 표시되지 않습니다**
A: 네트워크 연결과 API 서버 상태를 확인하세요. 브라우저 개발자 도구에서 오류 메시지를 확인할 수 있습니다.

**Q: 사용량 카드가 올바르지 않습니다**
A: Overview 섹션의 기간 필터를 확인하세요. 사용량 카드는 선택된 기간을 정확히 반영합니다.

### 기술적 문제

**Q: API 오류가 발생합니다**
A: 프록시 서버 포트(8001)와 대시보드 접속 URL 포트가 일치하는지 확인하세요.

**Q: 검색이 작동하지 않습니다**
A: 브라우저 캐시를 지우고 페이지를 새로고침하세요.

## 🔧 Git 작업 규칙

### 개발자 Git 작업 가이드
```
Git 작업은 엄격한 사용자 권한 규칙을 따라야 합니다:

Git 커밋 규칙:
- 코드 변경 완료 후 즉시 커밋
- 한국어로 설명적인 커밋 메시지 사용
- 커밋 메시지에 기술적 세부사항 포함
- 코드와 문서 변경사항을 함께 커밋

Git 푸시 규칙:
- 사용자가 명시적으로 요청하지 않으면 절대 원격 저장소에 푸시하지 않음
- 사용자가 구체적으로 원격 푸시를 요청할 때만 푸시
- 푸시 전에 항상 사용자와 확인
- 사용자의 명시적 승인을 받기 전까지 푸시 작업 대기

Git 브랜치 규칙:
- 사용자가 요청할 때만 새 브랜치 생성
- 설명적인 브랜치 이름 사용 (예: 날짜 기반 0803)
- 사용자가 요청할 때만 브랜치 전환
- 사용자가 푸시하기로 결정할 때까지 로컬 변경사항 유지

Git 워크플로우:
1. 코드 변경
2. 로컬에서 변경사항 커밋
3. 문서 업데이트
4. 사용자의 푸시 요청 대기
5. 그때만 원격 저장소에 푸시

사용자 권한 필요:
- 모든 원격 작업(push, pull, fetch)은 명시적 사용자 권한 필요
- 로컬 작업(commit, 브랜치 생성)은 자동으로 수행 가능
- 원격 저장소 상호작용 전에 항상 사용자에게 확인
```
